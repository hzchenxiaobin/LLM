# CUDA Top-K 性能基准测试

基于 RTX 5090 (Blackwell 架构) 优化教程的不同 CUDA Top-K 实现的综合性能比较。

## 概述

本项目实现并基准测试了四种在大型数组中查找前 K 个元素的不同方法:

| 版本 | 方案 | 描述 |
|---------|----------|-------------|
| **V0** | Thrust 基线 | 使用 Thrust 库进行全局排序 (O(N log N)) |
| **V1** | 线程级 | 使用每线程局部 Top-K 数组的 Map-Reduce |
| **V2** | 共享内存 | 共享内存中的块级合并 |
| **V3** | Warp Shuffle | 使用 warp 原语的寄存器到寄存器通信 |

## 要求

- **CUDA Toolkit**: 11.0 或更高版本 (RTX 5090 建议 12.8+)
- **GPU**: 计算能力 7.0+ 的 NVIDIA GPU
- **操作系统**: Linux (支持 WSL2)
- **编译器**: 支持 C++17 的 nvcc

## 实现细节

### V0: Thrust 基线
- 使用 `thrust::sort` 配合自定义比较器
- O(N log N) 的完全排序
- 复制前 K 个元素到输出
- **优点**: 简单、正确
- **缺点**: 过度设计, 浪费内存带宽

### V1: 线程级局部 Top-K
- 每个线程在寄存器中维护有序的局部 Top-K 数组
- 对输入进行 grid-stride 循环
- 所有线程将中间结果写入全局内存
- 最终归约内核合并所有线程结果
- **优点**: 比完全排序好
- **缺点**: 高中间内存流量 (num_threads × K)

### V2: 块级共享内存
- 与 V1 类似, 但块内线程在共享内存中合并
- 块只向全局内存写入 K 个元素
- 共享内存中的树形归约
- **优点**: 减少全局内存写入 (num_blocks × K vs num_threads × K)
- **缺点**: 仍然需要 `__syncthreads()` 同步

### V3: Warp Shuffle 原语
- 使用 `__shfl_down_sync` 进行寄存器到寄存器通信
- 先进行 warp 级归约, 然后 warp leader 写入共享内存
- 共享内存中使用 warp shuffle 进行最终归约
- **优点**: warp 级通信不需要共享内存, 延迟更低
- **缺点**: 更复杂的 warp 级编程

## 性能特征

RTX 5090 上的预期性能 (典型):

| 版本 | 时间 (10M, K=32) | 加速比 | 瓶颈 |
|---------|------------------|---------|------------|
| V0 | ~10-15 ms | 1.0x | 排序开销 |
| V1 | ~3-5 ms | 2-4x | 全局内存流量 |
| V2 | ~2-3 ms | 4-6x | 共享内存带宽 |
| V3 | ~1.5-2.5 ms | 5-8x | 指令吞吐量 |

## 优化技巧

### 针对 RTX 5090 (Blackwell)
- 使用每块 256 线程以获得最佳占用率
- 启动大量块 (128-256) 以使 SM 饱和
- 利用大 L2 缓存存储中间数据

### 通用指南
1. **小 K (≤ 8)**: V2 或 V3 最优
2. **中 K (8-64)**: V3 配合 warp shuffle 表现出色
3. **大 K (64-256)**: 考虑 Radix Select (V4, 未实现)

## 故障排除

### 构建错误
- **"No CUDA-capable device"**: 确保已安装 NVIDIA 驱动
- **"nvcc not found"**: 添加 CUDA 到 PATH: `export PATH=/usr/local/cuda/bin:$PATH`
- **"sm_100 not supported"**: RTX 5090 需要 CUDA 12.8+

### 运行时错误
- **显存不足**: 减小 N 或使用更小的 K
- **结果不正确**: 检查输入数据中是否有 NaN/Inf
