# CUDA Reduction V1: Interleaved Addressing 详解

## 代码概览

```cuda
__global__ void reduce_v1(float *g_idata, float *g_odata, unsigned int n) {
    extern __shared__ float sdata[];

    unsigned int tid = threadIdx.x;
    unsigned int i = blockIdx.x * blockDim.x + threadIdx.x;

    sdata[tid] = (i < n) ? g_idata[i] : 0.0f;
    __syncthreads();

    for (unsigned int s = 1; s < blockDim.x; s *= 2) {
        if (tid % (2 * s) == 0) {
            sdata[tid] += sdata[tid + s];
            }
        __syncthreads();
    }

    if (tid == 0) {
        atomicAdd(g_odata, sdata[0]);
    }
}
```

---

## 核心概念：Reduction（归约）

**Reduction** 是一种并行计算模式，将数组中的所有元素通过某个二元运算（如求和、求最大/最小值）归约为一个单一结果。

**示例**：`[1, 2, 3, 4, 5, 6, 7, 8]` → 求和 → `36`

---

## 代码逐段解析

### 1. 数据加载到共享内存

```cuda
extern __shared__ float sdata[];           // 动态分配共享内存
unsigned int tid = threadIdx.x;            // 线程在 block 内的索引
unsigned int i = blockIdx.x * blockDim.x + threadIdx.x;  // 全局数据索引

sdata[tid] = (i < n) ? g_idata[i] : 0.0f;  // 从全局内存加载到共享内存
__syncthreads();                           // 等待所有线程完成加载
```

**说明**：
- `sdata[]`: 共享内存，同一块内的所有线程都可以快速访问
- 每个线程负责加载一个元素到共享内存
- 边界检查：如果索引 `i >= n`，则存入 0（避免越界）

---

### 2. 核心 Reduction 循环

```cuda
for (unsigned int s = 1; s < blockDim.x; s *= 2) {
    if (tid % (2 * s) == 0) {    // 关键：间隔寻址条件
        sdata[tid] += sdata[tid + s];  // 相邻元素相加
    }
    __syncthreads();             // 每轮归约后同步
}
```

## 深入解析：`atomicAdd` 的作用

### 1. 作用概述

`atomicAdd(g_odata, sdata[0])` 的作用是**将当前 block 的归约结果（部分和）原子地累加到全局输出内存中**。

### 2. 为什么需要 atomicAdd？

**问题背景**：
- 在 CUDA 归约中，通常会启动多个 block 来处理大数组的不同部分
- 每个 block 独立完成归约后，得到该数据段的部分和（存储在 `sdata[0]`）
- 这些部分和需要汇总成一个最终的全局总和
- 由于多个 block 可能**同时完成并尝试写入 `g_odata`**，存在**竞争条件（Race Condition）**

**如果不使用 atomicAdd**：

```
假设 3 个 block 同时执行完成：
Block 0: 读取 g_odata = 0, 准备写入 0 + 10 = 10
Block 1: 读取 g_odata = 0, 准备写入 0 + 20 = 20  ← 读取到了旧值！
Block 2: 读取 g_odata = 0, 准备写入 0 + 30 = 30  ← 读取到了旧值！

最终结果可能是 10、20 或 30，而不是正确的 60
```

### 3. atomicAdd 的工作原理

`atomicAdd(float* address, float value)` 执行以下操作**原子地**（不可中断）：

1. 读取 `address` 指向的当前值
2. 将当前值与 `value` 相加
3. 将结果写回 `address`

```
Block 0: atomicAdd(g_odata, 10) → g_odata = 0 + 10 = 10
Block 1: atomicAdd(g_odata, 20) → g_odata = 10 + 20 = 30（读取到 Block 0 写入后的值）
Block 2: atomicAdd(g_odata, 30) → g_odata = 30 + 30 = 60（读取到 Block 1 写入后的值）

最终结果 = 60 ✓
```

### 5. 关键点总结

| 特性 | 说明 |
|------|------|
| **为什么只有 tid == 0 执行** | 每个 block 的归约结果只保存在 `sdata[0]`，只有线程 0 拥有最终结果 |
| **为什么是 atomic 操作** | 防止多个 block 同时写入全局内存时的竞争条件 |
| **性能影响** | atomic 操作有开销，但在最终阶段只执行一次（每个 block 一次），影响较小 |
| **替代方案** | 可以使用递归归约（多级 kernel 调用）或 Thrust 库避免显式 atomic 操作 |

## 算法为什么有效？

### 数学原理

每次迭代将数据量减少一半（折半归约）：
- 第 1 轮后：N 个元素 → N/2 个部分和
- 第 2 轮后：N/2 个元素 → N/4 个部分和
- ...
- 第 log₂(N) 轮后：1 个最终结果

**总工作量**：N + N/2 + N/4 + ... + 1 ≈ 2N = O(N)

---

### 为什么使用 `tid % (2*s) == 0`？

这是**间隔寻址（Interleaved Addressing）**的核心：

```
s=1: tid % 2 == 0  → 位置 0, 2, 4, 6 的线程工作
s=2: tid % 4 == 0  → 位置 0, 4 的线程工作  
s=4: tid % 8 == 0  → 位置 0 的线程工作
```

这种模式使得每轮归约时，工作线程均匀分布在剩余的数据位置上。

---

## 问题：Warp Divergence（线程束发散）

### 问题说明

这是 V1 版本的主要缺陷。在 GPU 中，线程以 **Warp**（32 个线程）为单位执行。Warp 内的所有线程必须执行相同的指令。

```
第 1 轮 (s=1): tid % 2 == 0
线程状态: ■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■
         工作 空闲 工作 空闲 工作 空闲 ...
         
Warp 执行: 一半线程工作，一半空闲，但必须一起执行
          → 性能损失！

第 2 轮 (s=2): tid % 4 == 0  
线程状态: ■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■
         工作 空闲 空闲 空闲 工作 空闲 ...
         
Warp 执行: 四分之一线程工作，四分之三空闲
          → 性能损失加剧！
```

### 发散程度统计

| 轮次 | 活跃线程比例 | 空闲线程比例 | Warp 效率 |
|------|-------------|-------------|----------|
| 1    | 50%         | 50%         | 50%      |
| 2    | 25%         | 75%         | 25%      |
| 3    | 12.5%       | 87.5%       | 12.5%    |
| ...  | ...         | ...         | ...      |

## 总结

| 特性 | 说明 |
|------|------|
| **算法名称** | Interleaved Addressing（间隔寻址） |
| **核心思想** | 相邻元素配对相加，迭代折半 |
| **时间复杂度** | O(log N) 并行时间 |
| **主要问题** | Warp Divergence（线程束发散） |
| **改进方向** | Strided Addressing（V2 版本解决） |

### 为什么这是"朴素版本"

虽然算法逻辑正确，但由于 Warp Divergence，实际 GPU 利用率很低。后续版本通过改变寻址模式（Sequential Addressing）来解决这个问题。
