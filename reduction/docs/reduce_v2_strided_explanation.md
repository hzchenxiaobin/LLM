# CUDA Reduction V2: Strided Addressing 详解

## 代码概览

```cuda
__global__ void reduce_v2(float *g_idata, float *g_odata, unsigned int n) {
    extern __shared__ float sdata[];
    unsigned int tid = threadIdx.x;
    unsigned int i = blockIdx.x * blockDim.x + threadIdx.x;

    sdata[tid] = (i < n) ? g_idata[i] : 0.0f;
    __syncthreads();

    for (unsigned int s = 1; s < blockDim.x; s *= 2) {
        int index = 2 * s * tid;
        if (index < blockDim.x) {
            sdata[index] += sdata[index + s];
        }
        __syncthreads();
    }

    if (tid == 0) atomicAdd(g_odata, sdata[0]);
}
```

---

## 核心改进：解决 Warp Divergence

### V1 vs V2 关键区别

| 特性 | V1 (Interleaved) | V2 (Strided) |
|------|------------------|--------------|
| **寻址模式** | `tid % (2*s) == 0` | `index = 2*s*tid` |
| **活跃线程分布** | 分散在 Warp 中 | 连续的前半部分 |
| **Warp Divergence** | 严重 | 消除 |
| **新问题** | - | Bank Conflict |

---

## 代码逐段解析

### 1. 数据加载（与 V1 相同）

```cuda
extern __shared__ float sdata[];
unsigned int tid = threadIdx.x;
unsigned int i = blockIdx.x * blockDim.x + threadIdx.x;

sdata[tid] = (i < n) ? g_idata[i] : 0.0f;
__syncthreads();
```

### 2. 核心改进：Sequential Addressing

```cuda
for (unsigned int s = 1; s < blockDim.x; s *= 2) {
    int index = 2 * s * tid;        // 关键：计算目标索引
    if (index < blockDim.x) {       // 只有前半部分线程参与
        sdata[index] += sdata[index + s];
    }
    __syncthreads();
}
```

**核心变化**：
- **V1**: `if (tid % (2*s) == 0)` → 线程分散活跃
- **V2**: `if (index < blockDim.x)` → 连续的前半部分线程活跃

## 为什么消除了 Warp Divergence？

### Warp 执行模型

在 GPU 中，一个 Warp 包含 32 个线程，它们**必须执行相同的指令**。如果 Warp 内的线程走不同分支，就会产生 Divergence。

### V1 vs V2 对比

**V1 (Interleaved) - 发散严重**：
```
s=1: tid % 2 == 0
Warp 线程状态: 工作 空闲 工作 空闲 工作 空闲 ...
              ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □ ■ □
              
问题: 一个 Warp 内同时有工作和空闲的线程
     → GPU 必须串行执行两个分支！
```

**V2 (Strided) - 无发散**：
```
s=1: index = 2*tid < blockDim.x → tid < 4
Warp 线程状态: 工作 工作 工作 工作 空闲 空闲 空闲 空闲 ...
              ■ ■ ■ ■ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □ □
              
优势: 整个 Warp 走相同分支（都是工作），或者整个 Warp 走另一个分支（都是空闲）
     → 没有 Divergence！
```

## 新问题：Bank Conflict（存储体冲突）

### 什么是 Bank Conflict？

共享内存被划分为 32 个 **Bank**（存储体）。如果同一 Warp 内的多个线程同时访问**同一个 Bank** 的不同地址，就会产生冲突，访问会被串行化。

### V2 中的 Bank Conflict

在 V2 中，`sdata[index]` 和 `sdata[index + s]` 的访问模式会导致冲突：

```
假设 bank 数 = 4（简化说明），每个 float 占 1 个 bank

第 1 轮 (s=1):
┌─────────────────────────────────────────────────────────────┐
│ Bank 0 │ Bank 1 │ Bank 2 │ Bank 3 │ Bank 0 │ Bank 1 │ Bank 2 │ Bank 3 │
│   0    │   1    │   2    │   3    │   4    │   5    │   6    │   7    │
│   ↑         ↑         ↑         ↑    
│  tid0      tid1      tid2      tid3   都在读取 bank 0！
│ (sdata[0]) (sdata[2]) (sdata[4]) (sdata[6])
└─────────────────────────────────────────────────────────────┘

同时：
┌─────────────────────────────────────────────────────────────┐
│ Bank 0 │ Bank 1 │ Bank 2 │ Bank 3 │ Bank 0 │ Bank 1 │ Bank 2 │ Bank 3 │
│   0    │   1    │   2    │   3    │   4    │   5    │   6    │   7    │
│             ↑         ↑         ↑         ↑
│            tid0      tid1      tid2      tid3   都在读取 bank 1！
│          (sdata[1]) (sdata[3]) (sdata[5]) (sdata[7])
└─────────────────────────────────────────────────────────────┘
```

**冲突分析**：
- `sdata[0]` 和 `sdata[4]` 都在 Bank 0
- `sdata[2]` 和 `sdata[6]` 都在 Bank 2
- 这导致 2-way bank conflict

### 冲突程度统计

| 轮次 | 活跃线程 | 访问模式 | Bank Conflict 程度 |
|------|---------|---------|-------------------|
| s=1 | 4 线程 | `sdata[2*tid] += sdata[2*tid+1]` | 2-way |
| s=2 | 2 线程 | `sdata[4*tid] += sdata[4*tid+2]` | 1-way（无冲突）|
| s=4 | 1 线程 | `sdata[8*tid] += sdata[8*tid+4]` | 无 |

## 性能对比总结

| 指标 | V1 (Interleaved) | V2 (Strided) |
|------|------------------|--------------|
| **Warp Divergence** | ❌ 严重（50% → 12.5%） | ✅ 消除 |
| **Bank Conflict** | ✅ 无 | ❌ 存在（2-way） |
| **整体性能** | 较差 | 较好（但仍有提升空间） |

### V2 的优势

1. **消除了 Warp Divergence**：活跃线程连续排列，Warp 内线程执行相同分支
2. **更好的指令吞吐量**：减少分支指令的开销
3. **更简单的条件判断**：`index < blockDim.x` 比 `tid % (2*s) == 0` 更高效

### V2 的劣势

1. **引入 Bank Conflict**：同一轮中多个线程访问相同 bank
2. **仍有优化空间**：后续版本（V3）将进一步解决 Bank Conflict

---

## 总结

| 特性 | 说明 |
|------|------|
| **算法名称** | Strided Addressing（顺序寻址） |
| **核心改进** | 用 `index = 2*s*tid` 替代 `tid % (2*s) == 0` |
| **主要优势** | 消除 Warp Divergence |
| **引入问题** | Bank Conflict |
| **改进方向** | 改变共享内存布局（V3 版本解决） |

V2 通过改变线程的活跃模式，成功解决了 V1 的 Warp Divergence 问题，但代价是引入了 Bank Conflict。这是 CUDA 优化的经典权衡：**解决一个问题往往会引入新问题**。
