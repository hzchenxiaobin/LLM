| 参数 | RTX 5090 数值 | RTX 4090 (对比) | 提升倍数 |
|:---|:---:|:---:|:---:|
| **架构** | Blackwell (GB202) | Ada Lovelace | - |
| **制程** | 5nm | 4nm | - |
| **SM 数量** | 170 | 128 | 1.33× |
| **CUDA Cores** | 21,760 | 16,384 | 1.33× |
| **Tensor Cores** | 680 (第 5 代) | 512 (第 4 代) | 1.33× |
| **RT Cores** | 170 | 128 | 1.33× |
| **显存** | 32 GB GDDR7 | 24 GB GDDR6X | 1.33× |
| **显存位宽** | 512-bit | 384-bit | 1.33× |
| **显存带宽** | **1,792 GB/s** | 1,008 GB/s | **1.78×** |
| **峰值算力 (FP32)** | **104.9 TFLOPS** | 82.6 TFLOPS | **1.27×** |
| **峰值算力 (FP16 Tensor)** | **835.3 TFLOPS** | 330.3 TFLOPS | **2.53×** |
| **TGP** | 575W | 450W | - |

## 2. SM (Streaming Multiprocessor) 架构

### 2.1 SM 整体结构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     Streaming Multiprocessor (SM)                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                     CUDA Core 阵列 (128个)                      │    │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐     ┌─────────┐          │    │
│  │  │FP32 Core│ │FP32 Core│ │FP32 Core│ ... │FP32 Core│          │    │
│  │  │  (FMA)  │ │  (FMA)  │ │  (FMA)  │     │  (FMA)  │          │    │
│  │  └─────────┘ └─────────┘ └─────────┘     └─────────┘          │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                          │
│  ┌─────────────────────────┐  ┌─────────────────────────────────────┐   │
│  │    Tensor Core (4个)     │  │         Warp 调度器 (4个)           │   │
│  │  ┌─────┐ ┌─────┐       │  │  ┌────────┐ ┌────────┐ ┌────────┐  │   │
│  │  │TC 0 │ │TC 1 │ ...    │  │  │Warp 0  │ │Warp 1  │ │Warp 2  │  │   │
│  │  │5th  │ │5th  │       │  │  │调度器  │ │调度器  │ │调度器  │  │   │
│  │  └─────┘ └─────┘       │  │  │        │ │        │ │        │  │   │
│  └─────────────────────────┘  └─────────────────────────────────────┘   │
│                                                                          │
│  ┌─────────────────────────┐  ┌─────────────────────────────────────┐   │
│  │    寄存器文件 (256 KB)    │  │     共享内存 / L1 (164 KB)          │   │
│  │    65,536 个 32-bit      │  │     可配置分配比例                   │   │
│  └─────────────────────────┘  └─────────────────────────────────────┘   │
│                                                                          │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                     Load/Store 单元 (32个)                        │    │
│  │                    特殊函数单元 SFU (32个)                        │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 SM 核心组件规格

| 组件 | 每 SM 数量 | 功能说明 |
|:---|:---:|:---|
| **FP32 CUDA Core** | 128 | 单精度浮点 FMA，基础计算单元 |
| **FP64 CUDA Core** | 4 | 双精度浮点 (1:32 比例) |
| **INT32 Core** | 64 | 32位整数运算 |
| **Tensor Core** | 4 | 第 5 代，支持 FP16/BF16/FP8/FP4 |
| **Load/Store Unit** | 32 | 内存加载/存储 |
| **SFU** | 32 | 特殊函数 (sin/cos/sqrt) |
| **Warp 调度器** | 4 | 每周期发射 4 条 Warp 指令 |
| **寄存器文件** | 256 KB | 65,536 个 32-bit 寄存器 |
| **共享内存** | 164 KB | 与 L1 缓存共享，可配置 |

### 2.3 Block 与 SM 的关系

```
关键约束：
┌─────────────────────────────────────────────────────────┐
│  一个 Block 只能在一个 SM 上执行                          │
│  一个 SM 可以同时执行多个 Block（资源允许时）              │
│  Block 之间无法直接同步或通信                             │
└─────────────────────────────────────────────────────────┘

资源分配示例：

SM 0:
  ┌───────────────┐ ┌───────────────┐
  │ Block (0, 0)  │ │ Block (1, 0)  │
  │ 256 线程      │ │ 256 线程      │
  │ 8 KB SMEM     │ │ 8 KB SMEM     │
  │ 20 KB Reg     │ │ 20 KB Reg     │
  └───────────────┘ └───────────────┘
  ↑ 时间片轮转执行

SM 1:
  ┌───────────────┐
  │ Block (0, 1)  │
  │ 256 线程      │
  │ 8 KB SMEM     │
  └───────────────┘
```

## 3. 寄存器文件与优化

### 3.1 寄存器文件架构

```
寄存器文件组织结构：

┌─────────────────────────────────────────────────────────┐
│              每 SM 256 KB 寄存器文件                       │
│           = 65,536 个 32-bit (4-byte) 寄存器              │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌─────────────┐ ┌─────────────┐     ┌─────────────┐  │
│  │  Warp 0      │ │  Warp 1      │ ... │  Warp N      │  │
│  │  (32线程)    │ │  (32线程)    │     │  (32线程)    │  │
│  │  R0-R255    │ │  R0-R255    │     │  R0-R255    │  │
│  │  /thread     │ │  /thread     │     │  /thread     │  │
│  └─────────────┘ └─────────────┘     └─────────────┘  │
│                                                          │
│  每线程最多 255 个寄存器                                  │
│  每 Warp 最多 255 × 32 = 8,160 个寄存器                   │
│  每 SM 最多 65,536 / (255 × 32) ≈ 8 Warps（100% 寄存器） │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Occupancy（占用率）计算

Occupancy = 实际运行的 Warp 数 / SM 支持的最大 Warp 数

```
RTX 5090 每 SM 限制：
- 最大 Warp 数：64 (2048 线程)
- 最大 Block 数：32

Occupancy 计算公式：
┌─────────────────────────────────────────────────────────┐
│ 实际 Warp 数 = min(                                      │
│   寄存器限制: floor(65536 / (R × 32))                   │
│   共享内存限制: floor(164KB / SMEM_per_block) × warps_per_block │
│   硬件限制: 64                                            │
│ )                                                        │
└─────────────────────────────────────────────────────────┘
```

**Occupancy 与寄存器使用关系表**：

| 每线程寄存器数 | 每 Warp 占用 | 每 SM 最大 Warp | Occupancy | 适用场景 |
|:---:|:---:|:---:|:---:|:---|
| 32 | 4 KB | 64 | 100% | 内存密集型 Kernel |
| 64 | 8 KB | 32 | 50% | 通用计算 |
| 96 | 12 KB | 21 | 33% | 平衡型 |
| **128** | 16 KB | 16 | **25%** | **GEMM 推荐** |
| 160 | 20 KB | 12 | 19% | 计算密集型 |
| 192 | 24 KB | 10 | 16% | 高 ILP 需求 |
| 255 | ~32 KB | 8 | 12.5% | 极限寄存器使用 |

### 3.3 GEMM 寄存器使用分析

```cuda
// sgemm_register.cu 的寄存器使用
float accum[TM][TN];    // accum[8][8] = 64 个寄存器
float frag_a[TM];        // frag_a[8]    = 8 个寄存器
float frag_b[TN];        // frag_b[8]    = 8 个寄存器
// 其他临时变量          ~10 个寄存器
// 总计                  ~90 个寄存器/线程
```

**计算验证**：
```
每线程: 90 个寄存器
每 Warp: 90 × 32 = 2,880 个寄存器 = ~11.25 KB
每 Block (256 线程): 90 × 256 = 23,040 个寄存器
每 Block Warp 数: 256 / 32 = 8 Warps

每 SM 可同时运行: 256 KB / 11.25 KB ≈ 22 Warps
考虑 Block 限制: 22 / 8 = 2.75 个 Block
实际并发 Block 数: 2
实际并发 Warp 数: 2 × 8 = 16
Occupancy: 16 / 64 = 25%
```

### 3.4 寄存器优化策略

```
GEMM 的 Occupancy 策略：

目标 Occupancy：25%-50%（而非 100%）

原因：
1. GEMM 是计算密集型，主要瓶颈是计算单元利用率
2. 使用更多寄存器（降低 Occupancy）可以提高 ILP
3. 更多寄存器 = 更大的 Thread Tile = 更高的计算/访存比

优化建议：
- 目标：每线程 64-128 个寄存器
- 对应 Occupancy：25%-50%
- 平衡 ILP 和延迟隐藏
```

## 4. 共享内存架构

### 4.1 共享内存物理结构

```
共享内存 Bank 结构：

┌─────────────────────────────────────────────────────────┐
│                   共享内存物理布局                         │
├─────────────────────────────────────────────────────────┤
│  Bank 0   Bank 1   Bank 2   ...  Bank 30   Bank 31      │
│  (128B)   (128B)   (128B)        (128B)   (128B)       │
├─────────────────────────────────────────────────────────┤
│  Word 0   Word 1   Word 2   ...  Word 30   Word 31      │
│  [0x00]   [0x04]   [0x08]        [0x78]   [0x7C]       │
│  Word 32  Word 33  Word 34  ...  Word 62   Word 63      │
│  [0x80]   [0x84]   [0x88]        [0xF8]   [0xFC]       │
│  ...                                                    │
└─────────────────────────────────────────────────────────┘

关键参数：
- Bank 数量：32 (与 Warp 大小匹配)
- 每个 Bank 宽度：32 bits (4 bytes for float)
- 地址映射公式：Bank Index = (address / 4) % 32
```

### 4.2 Bank Conflict 原理

**理想情况**（无 Conflict）：
```
Warp 中的 32 个线程同时访问：

Thread 0 → Bank 0    ┐
Thread 1 → Bank 1    │
Thread 2 → Bank 2    │  全部并行
...                  │  1 个周期完成
Thread 31 → Bank 31  ┘

总耗时 = 1 cycle
带宽利用率 = 100%
```

**Conflict 情况**（8-way Conflict）：
```
Warp 中的 32 个线程同时访问：

Thread 0-3  → Bank 0   ┐
Thread 4-7  → Bank 1   │
Thread 8-11 → Bank 2   │  串行执行
...                    │  需要 4 个周期
Thread 28-31→ Bank 7   ┘

总耗时 = 4 cycles
带宽利用率 = 25%
```

### 4.3 Bank Conflict 消除：Padding

```
原始布局（有 Conflict）：
__shared__ float sA[128][8];

行 0: 起始地址 0,    Bank = (0/4)%32 = 0,   占用 Banks 0-7
行 8: 起始地址 256,  Bank = (256/4)%32 = 0, 占用 Banks 0-7  ← 与行 0 冲突！

Padding 后（无 Conflict）：
__shared__ float sA[128][9];  // +1 Padding

行 0: 起始地址 0,    Bank = (0/4)%32 = 0,   占用 Banks 0-7
行 8: 起始地址 288,  Bank = (288/4)%32 = 8, 占用 Banks 8-15 ← 不冲突！
```

## 5. Tensor Cores 详解

### 5.1 Tensor Core 架构

```
第 5 代 Tensor Core 结构：

┌─────────────────────────────────────────────────────────┐
│                   Tensor Core (5th Gen)                 │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  支持的计算精度：                                          │
│  ┌────────┬────────┬────────┬────────┬────────┐        │
│  │  FP16  │  BF16  │  FP8   │  FP4   │  TF32  │        │
│  │(16-bit)│(16-bit)│(8-bit) │(4-bit) │(19-bit)│        │
│  └────────┴────────┴────────┴────────┴────────┘        │
│                                                          │
│  基本计算单元：                                            │
│  ┌─────────────────────────────────────────────┐        │
│  │  D = A × B + C                               │        │
│  │  A: 8×4×16  (M×K×bits)                       │        │
│  │  B: 4×8×16  (K×N×bits)                       │        │
│  │  C/D: 8×8×32 (M×N×bits)                      │        │
│  │  一次操作 = 128 FMA = 256 FLOPS (FP16)       │        │
│  └─────────────────────────────────────────────┘        │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 5.2 Tensor Core vs CUDA Core 性能对比

| 计算单元 | 每周期操作 | 单 SM 峰值 | 全芯片峰值 |
|:---|:---:|:---:|:---:|
| **CUDA Core (FP32)** | 2 FLOPS (FMA) | 256 FLOPS/cycle | ~104.9 TFLOPS |
| **Tensor Core (FP16)** | 256 FLOPS | 1,024 FLOPS/cycle | **835.3 TFLOPS** |
| **Tensor Core (FP8)** | 512 FLOPS | 2,048 FLOPS/cycle | **1,670.6 TFLOPS** |

**关键洞察**：
```
Tensor Core FP16 性能 = 8 × CUDA Core FP32 性能

GEMM 优化目标：
- 基础优化 (CUDA Core): 30-50 TFLOPS (~30-50%)
- Tensor Core 优化: 200+ TFLOPS FP16 (~25%)
- 极限优化 (CUTLASS): 400+ TFLOPS FP8 (~50%)
```

### 5.3 WMMA API 使用示例

```cuda
#include <mma.h>
using namespace nvcuda;

__global__ void wmma_gemm(half *A, half *B, float *C, int M, int N, int K) {
    // 定义 WMMA 矩阵片段
    // 16x16x16 是常用的 Tensor Core 形状
    wmma::fragment<wmma::matrix_a, 16, 16, 16, half, wmma::row_major> a_frag;
    wmma::fragment<wmma::matrix_b, 16, 16, 16, half, wmma::col_major> b_frag;
    wmma::fragment<wmma::accumulator, 16, 16, 16, float> c_frag;

    // 初始化累加器
    wmma::fill_fragment(c_frag, 0.0f);

    // 计算当前 Warp 的坐标
    int warpM = (blockIdx.x * blockDim.x + threadIdx.x) / 32;
    int warpN = (blockIdx.y * blockDim.y + threadIdx.y);

    // K 维度循环
    for (int i = 0; i < K; i += 16) {
        // 加载矩阵块到 Fragment
        wmma::load_matrix_sync(a_frag, A + warpM * 16 * K + i, K);
        wmma::load_matrix_sync(b_frag, B + i * N + warpN * 16, N);

        // 执行 Tensor Core 矩阵乘法
        wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);
    }

    // 写回结果
    wmma::store_matrix_sync(C + warpM * 16 * N + warpN * 16, c_frag, N, 
                            wmma::mem_row_major);
}
```

## 6. 性能峰值与 Roofline 分析

### 6.1 理论峰值计算

#### FP32 CUDA Core 峰值

```
公式: Peak FLOPS = 2 × SM 数 × 每 SM CUDA Core 数 × 核心频率

RTX 5090:
- SM 数量: 170
- 每 SM FP32 Core: 128
- 核心频率: ~2.41 GHz (Boost)
- FMA = 2 FLOPS (乘 + 加)

计算:
Peak = 2 × 170 × 128 × 2.41×10^9
     = 104.88 × 10^12 FLOPS
     = 104.9 TFLOPS
```

#### 显存带宽峰值

```
GDDR7 显存带宽计算：
- 等效频率: 28 Gbps (GDDR7)
- 位宽: 512-bit = 64 bytes
- 通道数: 1 (GDDR7 为单通道架构)

带宽 = 28×10^9 × 64 / 8
     = 1,792 GB/s
```

### 6.2 Roofline 模型

```
Roofline 模型图示：

Performance (TFLOPS)
    │
105 ┤───────────────────────────────┐ ← 计算屋顶 (Peak FLOPS)
    │                               │
 70 ┤                               │
    │                          ○ Tensor Core
 50 ┤                    ┌────────┘
    │               ┌────┘
 30 ┤          ┌────┘
    │     ┌────┘
 10 ┤─────┘ ← 内存屋顶斜线
    │    /
    │   / Memory Bandwidth = 1,792 GB/s
    │  /
    │ /
    ┼/──────────────────────────────────────
     1   10   58.5   100  1000  → Arithmetic Intensity (FLOPs/byte)
              ↑
         Ridge Point

Ridge Point = Peak FLOPS / Memory Bandwidth
            = 104.9 / 1.792
            = 58.5 FLOPs/byte

解读：
- AI < 58.5: 内存受限（Memory Bound）
- AI > 58.5: 计算受限（Compute Bound）
```

### 6.3 GEMM 的 Roofline 位置

| Kernel 类型 | Arithmetic Intensity | 瓶颈类型 | 理论利用率 |
|:---|:---:|:---:|:---:|
| **Naive** | ~0.5 | 内存带宽 | ~0.5% |
| **Shared Memory** | ~683 | 计算/共享内存 | ~9% |
| **Register Tiling** | ~683 | 计算 | ~30-50% |
| **Tensor Core** | ~100+ | 计算 | ~60-80% |

```
Shared Memory Tiling 提升：
AI 从 0.5 → 683，跨越 Ridge Point (58.5)
从内存受限 → 计算受限
性能从 ~1 TFLOPS → ~30-50 TFLOPS
```

## 7. 硬件优化检查清单

### GEMM Kernel 设计检查表

| 检查项 | 目标值 | 说明 |
|:---|:---:|:---|
| **寄存器使用** | ≤ 128/线程 | 保持 25-50% Occupancy |
| **共享内存使用** | ≤ 48 KB/Block | 允许 3-5 Blocks 并发 |
| **Bank Conflict** | 0 | 使用 Padding 消除 |
| **Block 大小** | 128-256 线程 | 4-8 Warps |
| **Thread Tile** | 8×8 或更大 | 提高计算/访存比 |
| **Occupancy** | 25-50% | 平衡 ILP 和延迟隐藏 |
