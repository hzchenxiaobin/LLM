# V4: ILP 与 Block 形状优化详解

## 概述

这是 CUDA 矩阵转置的**第四代优化版本**，核心改进是通过 **ILP（指令级并行）** 和 **Block 形状优化** 进一步提升性能，更好地隐藏内存延迟并提高 GPU 利用率。

---

## 完整代码

```cpp
#include "transpose_common.h"

#define TILE_DIM    32
#define BLOCK_ROWS   8  // 32 / 8 = 4，每线程处理 4 个元素

// v4: ILP 与 Block 形状优化
// 提高指令级并行(ILP)，每个线程处理多个元素
// 256线程/block可以更好地隐藏延迟，提高occupancy
__global__ void transpose_optimized_kernel(const float *A, float *B, int N) {
    __shared__ float tile[TILE_DIM][TILE_DIM + 1];  // 使用padding

    int x = blockIdx.x * TILE_DIM + threadIdx.x;
    int y = blockIdx.y * TILE_DIM + threadIdx.y;

    // 1. 每个线程读取 4 个元素 (ILP=4)
    for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
        if (x < N && (y + j) < N) {
            tile[threadIdx.y + j][threadIdx.x] = A[(y + j) * N + x];
        }
    }
    __syncthreads();

    // 2. 互换 Block 坐标
    int bx = blockIdx.y * TILE_DIM + threadIdx.x;
    int by = blockIdx.x * TILE_DIM + threadIdx.y;

    // 3. 每个线程写入 4 个元素
    for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
        if (bx < N && (by + j) < N) {
            B[(by + j) * N + bx] = tile[threadIdx.x][threadIdx.y + j];
        }
    }
}

void transpose_optimized(const float *d_A, float *d_B, int N, cudaStream_t stream) {
    dim3 block(TILE_DIM, BLOCK_ROWS);  // 32 x 8 = 256 线程
    dim3 grid((N + TILE_DIM - 1) / TILE_DIM, (N + TILE_DIM - 1) / TILE_DIM);
    transpose_optimized_kernel<<<grid, block, 0, stream>>>(d_A, d_B, N);
}
```

## 核心改进点

### 1. 线程块形状改变

```cpp
// V3 版本
#define TILE_DIM 32
dim3 block(TILE_DIM, TILE_DIM);  // 32 × 32 = 1024 线程/Block

// V4 版本
#define TILE_DIM    32
#define BLOCK_ROWS   8
dim3 block(TILE_DIM, BLOCK_ROWS);  // 32 × 8 = 256 线程/Block
```

**为什么从 1024 线程降到 256 线程？**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    GPU Streaming Multiprocessor (SM)                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   RTX 5090 (Blackwell) 每个 SM 的资源限制：                         │
│   • 每个 SM 最大线程数：1536 或 2048 (架构相关)                      │
│   • 每个 SM 最大 Block 数：32                                      │
│   • 每个 Block 最大线程数：1024                                    │
│                                                                     │
│   V3 配置 (1024 线程/Block)：                                       │
│   ├─ 每个 SM 最多同时运行：1 或 2 个 Block                         │
│   ├─ Active Threads：1024 或 2048                                  │
│   └─ 问题：线程太多，资源竞争大，延迟隐藏能力受限                   │
│                                                                     │
│   V4 配置 (256 线程/Block)：                                        │
│   ├─ 每个 SM 最多同时运行：6 或 8 个 Block                         │
│   ├─ Active Threads：1536 或 2048 (达到上限)                       │
│   └─ 优势：更多 Block 可调度，更好地隐藏延迟                        │
│                                                                     │
│   关键指标：Occupancy (占用率)                                      │
│   • V3: 可能无法填满 SM，因为单个 Block 消耗资源过多               │
│   • V4: 更多小 Block 可以更好地填满 SM，提高并行度                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. ILP (Instruction Level Parallelism)

```cpp
// 每个线程处理 TILE_DIM / BLOCK_ROWS = 32 / 8 = 4 个元素
for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
    if (x < N && (y + j) < N) {
        tile[threadIdx.y + j][threadIdx.x] = A[(y + j) * N + x];
    }
}
```

**什么是 ILP？**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    ILP (指令级并行) 原理                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   传统方式 (V3)：每个线程处理 1 个元素                              │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  线程执行:                                                    │  │
│   │    1. 计算地址                                                │  │
│   │    2. 发送内存读取请求                                        │  │
│   │    3. 等待数据返回 (停顿!)                                    │  │
│   │    4. 处理数据                                                │  │
│   │    5. 发送内存写入请求                                        │  │
│   │    6. 等待完成 (停顿!)                                        │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   ILP 方式 (V4)：每个线程处理 4 个元素，交错执行                    │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  线程执行:                                                    │  │
│   │    1. 计算地址 1                                              │  │
│   │    2. 发送读取请求 1                                          │  │
│   │    3. 计算地址 2 (不等待!)                                    │  │
│   │    4. 发送读取请求 2                                          │  │
│   │    5. 计算地址 3 (不等待!)                                    │  │
│   │    6. 发送读取请求 3                                          │  │
│   │    7. 计算地址 4 (不等待!)                                    │  │
│   │    8. 发送读取请求 4                                          │  │
│   │    9. 数据 1 返回，处理 (前面请求已发出，隐藏了延迟)          │  │
│   │   10. 数据 2 返回，处理                                       │  │
│   │   ...                                                         │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   效果：通过连续发出多个独立内存请求，隐藏单个请求的延迟            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 性能优化原理

### 1. 延迟隐藏 (Latency Hiding)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    内存延迟隐藏原理                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   GPU 内存访问延迟：数百个时钟周期                                    │
│                                                                     │
│   V3 (单元素处理)：                                                   │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │ [计算][读内存]████████████等待████████████[处理][写内存]等待 │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                               ↓                                     │
│                        大量时间浪费在等待上                           │
│                                                                     │
│   V4 (4元素 ILP)：                                                    │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │ [计算1][读1][计算2][读2][计算3][读3][计算4][读4]             │ │
│   │                      ↓ 读1完成                                │ │
│   │              [处理1][写1]读2完成→[处理2][写2]...               │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                               ↓                                     │
│                        计算与内存访问重叠                            │
│                                                                     │
│   原理：当一个 warp 等待内存时，GPU 可以：                            │
│   1. 执行其他就绪的 warp                                             │
│   2. 同一线程执行独立指令 (ILP)                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. Occupancy 提升

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Occupancy (占用率) 计算                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   假设 SM 资源限制：                                                  │
│   • 每个 SM 最大线程：2048                                           │
│   • 每个 SM 最大 Block：32                                           │
│   • 每个 Block 最大线程：1024                                        │
│                                                                     │
│   V3 (1024 线程/Block)：                                              │
│   ├─ 受限于 Block 最大线程数，1 个 Block 占 1024 线程                │
│   ├─ SM 可同时运行：min(2048/1024, 32) = 2 个 Block                │
│   ├─ Active Warps：2 × 32 = 64 Warps (假设全用)                    │
│   └─ 实际可能更低，因为共享内存等资源限制                            │
│                                                                     │
│   V4 (256 线程/Block)：                                               │
│   ├─ 1 个 Block 占 256 线程                                         │
│   ├─ SM 可同时运行：min(2048/256, 32) = 8 个 Block                   │
│   ├─ Active Warps：8 × 8 = 64 Warps                                 │
│   └─ 更多 Block 意味着更多独立的调度单元，更好的延迟隐藏            │
│                                                                     │
│   关键优势：                                                          │
│   • 更多 Block → 更多独立的 Tile 处理                                 │
│   • 一个 Block 等待内存时，其他 Block 可以继续计算                   │
│   • 整体吞吐量提升                                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 关键代码要点

### 1. 循环步长计算

```cpp
#define TILE_DIM    32
#define BLOCK_ROWS   8

// 循环次数 = TILE_DIM / BLOCK_ROWS = 32 / 8 = 4
for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
    // j = 0, 8, 16, 24
}
```

**为什么 `j += BLOCK_ROWS` 而非 `j++`？**
- 线程 `threadIdx.y` 负责处理行 `threadIdx.y`, `threadIdx.y+8`, `threadIdx.y+16`, `threadIdx.y+24`
- 步长为 8 确保 8 个线程组覆盖全部 32 行（0-7, 8-15, 16-23, 24-31）

### 2. 索引计算变化

```cpp
// V3 单元素
int y = blockIdx.y * TILE_DIM + threadIdx.y;  // 固定行
A[y * N + x];

// V4 多元素
int y = blockIdx.y * TILE_DIM + threadIdx.y;  // 起始行
for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
    A[(y + j) * N + x];  // 行随 j 变化
}
```

### 3. 边界条件处理

```cpp
if (x < N && (y + j) < N)  // 检查行边界
```
- 矩阵尺寸 N 不是 32 的倍数时需要边界检查
- `y + j` 确保每个处理的位置都在有效范围内

## 进一步优化建议

### 循环展开 (Loop Unrolling)

```cpp
// 编译器提示：手动展开循环
#pragma unroll
for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
    // ...
}

// 或手动展开：
tile[threadIdx.y + 0][threadIdx.x] = A[(y + 0) * N + x];
tile[threadIdx.y + 8][threadIdx.x] = A[(y + 8) * N + x];
tile[threadIdx.y + 16][threadIdx.x] = A[(y + 16) * N + x];
tile[threadIdx.y + 24][threadIdx.x] = A[(y + 24) * N + x];
```

**效果**：减少循环开销，允许更多指令重排优化。

### V4 的核心贡献

```
┌─────────────────────────────────────────────────────────────────────┐
│                         V4 优化总结                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. ILP (指令级并行)                                                 │
│     • 每个线程处理 4 个元素而非 1 个                                 │
│     • 连续发出多个内存请求，隐藏延迟                                 │
│     • 计算与内存访问重叠                                             │
│                                                                     │
│  2. Block 形状优化                                                    │
│     • 1024 线程/Block → 256 线程/Block                               │
│     • 每 SM 2 Block → 8 Block                                       │

│                                                                     │
│  3. 保留 V3 优势                                                      │
│     • Padding 消除 Bank Conflict                                     │
│     • 合并全局内存访问                                               │
│                                                                     │
│  性能提升: 60~70% (V3) → 70~80% (V4)                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 适用场景

- **内存密集型操作**（如转置、拷贝、归约）
- **高延迟内存访问**（全局内存）
- **计算量小、访存量大**的 kernel

### 后续优化方向

**V5 (v5_vectorized.cu)** 将向量化访问：
- 使用 `float4` 一次处理 16 字节
- 进一步减少内存指令数量
