代码实现

// Strided 循环：每个线程从 tid 开始，每次增加 blockDim.x
for (int i = tid; i < N; i += blockDim.x) {
    local_max = fmaxf(local_max, x[i]);
}
优点：
1. 内存访问合并（coalesced）- 相邻线程访问相邻内存
2. 负载均衡 - 自动处理 N 不是线程数倍的情况

归约代码详解

// 1. 每个线程写自己的 local_max 到共享内存
sdata[tid] = local_max;
__syncthreads();  SYNC

// 2. 二分归约：stride 从 256, 128, 64, ... 到 1
for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) {
    if (tid < stride) {
        // 当前线程比较自己与对面线程的值
        sdata[tid] = fmaxf(sdata[tid], sdata[tid + stride]);
    }
    __syncthreads();  SYNC
}

// 3. 结果在 sdata[0]
float row_max = sdata[0];
__syncthreads();  SYNC

关键改进：V2 消除了中间结果 (max_vals, sum_vals) 的全局内存读写， 全部改用共享内存，延迟从 ~400 cycles 降至 ~30 cycles。

七、优缺点总结

✅ 优点

  • 核函数融合 - 1 次 kernel launch，减少启动开销
  • 共享内存加速 - 中间结果低延迟访问 (~30 vs ~400 cycles)
  • 并行归约 - Tree reduction O(logN) 比串行 O(N) 高效
  • 内存访问减少 - 避免中间结果的全局内存读写
  • 协作式处理 - 多线程处理一行，适合 N 较大的情况

❌ 缺点

  • 同步开销 - 需要多次 __syncthreads()
  • 共享内存限制 - 每 block 2KB，限制并发 block 数
  • 线程利用率 - 归约后期很多线程空闲
  • bank conflict - 不当访问模式可能冲突
  • N 较小的情况 - N < 128 时部分线程完全空闲

八、适用场景

✅ 推荐使用：
  • N 较大（1024, 4096, 10000+）- 发挥多线程协作优势
  • M 中等（几百到几千行）- 足够 block 填充 GPU
  • 共享内存充足的现代 GPU（如 RTX 5090）
❌ 不推荐：
  • N 很小（< 128）- 大量线程空闲，归约开销占比大
  • M 很小（< 10）- 无法填满 GPU，并行度不足
  • 共享内存受限的旧 GPU

九、进一步优化方向

优化方向 描述 预期收益
Warp Shuffle 使用 __shfl_down_sync 替代 shared memory 进行 warp 级归约 减少共享内存使用，降低同步开销
向量化访存 使用 float4 一次读取 4 个 float 提升内存带宽利用率
寄存器优化 减少 shared memory 使用，改用 warp register 更低延迟，更多并发 block
持久化线程块 一个 block 处理多行 减少 block 总数，降低调度开销
计算与通信重叠 在读取数据时同时计算 隐藏内存延迟
