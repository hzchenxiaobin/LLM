# RTX 5090 Occupancy 计算练习题

本文档提供一系列 Occupancy 计算练习题，帮助你深入理解 GPU 资源限制和调度机制。

---

## 基础参数回顾

### RTX 5090 关键参数

| 参数 | 数值 |
|------|------|
| **SM 数量** | 170 |
| **每 SM 最大线程数** | 2048 |
| **每 SM 最大寄存器** | 256 KB = 262,144 bytes = 65,536 个 32-bit 寄存器 |
| **每 SM 最大共享内存** | 164 KB = 167,936 bytes |
| **每 Block 最大线程数** | 1024 |
| **每线程最大寄存器** | 255 |
| **Warp 大小** | 32 线程 |

### Occupancy 计算公式

```
Occupancy = 实际活跃 Warp 数 / 每 SM 最大 Warp 数

其中：
- 每 SM 最大 Warp 数 = 2048 / 32 = 64 Warps
- 实际活跃 Warp 数 = min(线程限制, 寄存器限制, 共享内存限制) / 32
```

## 练习题

### 题目 1：基础寄存器计算

**问题**：
你编写了一个 CUDA kernel，每个线程使用 64 个寄存器，Block 大小为 256 线程。

1. 计算每 SM 最多可以同时运行多少个 Block？
2. 计算此时的 Occupancy。

**提示**：
- 每线程使用 64 个 32-bit 寄存器 = 256 bytes/线程
- 每 Block 256 线程

<details>
<summary>点击查看答案</summary>

**解答**：

1. **每 SM 寄存器限制计算**：
   ```
   每线程寄存器 = 64 个 × 4 bytes = 256 bytes
   每 Block 需要 = 256 线程 × 256 bytes = 65,536 bytes = 64 KB
   
   每 SM 最大 Block 数 = 256 KB / 64 KB = 4 个 Block
   ```

2. **Occupancy 计算**：
   ```
   每 Block Warp 数 = 256 / 32 = 8 Warps
   活跃 Warp 总数 = 4 Block × 8 Warps = 32 Warps
   
   Occupancy = 32 / 64 = 50%
   ```

</details>

### 题目 2：共享内存限制

**问题**：
你编写了一个矩阵乘法 kernel，配置如下：
- Block 大小：32×32 = 1024 线程
- 共享内存使用：`__shared__ float sA[64][64]; __shared__ float sB[64][64];`

1. 计算该 kernel 使用的共享内存大小。
2. 计算每 SM 最多可以同时运行多少个 Block。
3. 计算此时的 Occupancy。

**提示**：
- sA[64][64] = 4096 个 float = 16 KB
- sB[64][64] = 4096 个 float = 16 KB

<details>
<summary>点击查看答案</summary>

**解答**：

1. **共享内存使用**：
   ```
   sA: 64 × 64 × 4 bytes = 16,384 bytes = 16 KB
   sB: 64 × 64 × 4 bytes = 16,384 bytes = 16 KB
   总计: 32 KB / Block
   ```

2. **每 SM 最大 Block 数**：
   ```
   每 SM 共享内存 = 164 KB
   每 Block 使用 = 32 KB
   最大 Block 数 = 164 / 32 = 5.125 → 5 个 Block
   
   同时检查线程限制：
   每 Block 线程 = 1024
   每 SM 最大线程 = 2048
   线程限制 Block 数 = 2048 / 1024 = 2 个 Block
   
   取最小值: min(5, 2) = 2 个 Block
   ```

3. **Occupancy 计算**：
   ```
   每 Block Warp 数 = 1024 / 32 = 32 Warps
   活跃 Warp 总数 = 2 × 32 = 64 Warps
   
   Occupancy = 64 / 64 = 100%
   ```

**注意**：虽然共享内存可以支持 5 个 Block，但线程限制只能支持 2 个 Block。实际并发受线程限制约束，Occupancy 为 100%。

</details>

### 题目 3：寄存器与共享内存双重限制

**问题**：
sgemm_register.cu 的配置如下：
- Block 大小：16×16 = 256 线程
- 寄存器使用：
  - `float accum[8][8]` = 64 个寄存器
  - `float frag_a[8]` = 8 个寄存器
  - `float frag_b[8]` = 8 个寄存器
  - 其他临时变量：约 10 个寄存器
- 共享内存使用：`__shared__ float sA[128][8]; __shared__ float sB[8][128];`

1. 计算每线程总寄存器使用数。
2. 计算每 Block 共享内存使用。
3. 计算每 SM 最大 Block 数（考虑寄存器和共享内存限制）。
4. 计算最终的 Occupancy。

<details>
<summary>点击查看答案</summary>

**解答**：

1. **每线程寄存器使用**：
   ```
   accum[8][8]: 64 个
   frag_a[8]: 8 个
   frag_b[8]: 8 个
   其他: 10 个
   总计: 90 个寄存器/线程
   ```

2. **每 Block 共享内存使用**：
   ```
   sA[128][8]: 128 × 8 × 4 = 4,096 bytes = 4 KB
   sB[8][128]: 8 × 128 × 4 = 4,096 bytes = 4 KB
   总计: 8 KB / Block
   ```

3. **每 SM 最大 Block 数**：

   **寄存器限制**：
   ```
   每线程寄存器 = 90 × 4 = 360 bytes
   每 Block 需要 = 256 × 360 = 92,160 bytes ≈ 90 KB
   
   每 SM 最大 Block 数 = 256 KB / 90 KB = 2.84 → 2 个 Block
   ```

   **共享内存限制**：
   ```
   每 SM 最大 Block 数 = 164 KB / 8 KB = 20.5 → 20 个 Block
   ```

   **线程限制**：
   ```
   每 SM 最大 Block 数 = 2048 / 256 = 8 个 Block
   ```

   **取最小值**：min(2, 20, 8) = **2 个 Block**

   **限制因素**：寄存器使用！

4. **Occupancy 计算**：
   ```
   每 Block Warp 数 = 256 / 32 = 8 Warps
   活跃 Warp 总数 = 2 × 8 = 16 Warps
   
   Occupancy = 16 / 64 = 25%
   ```

**分析**：寄存器使用是瓶颈！如果能将每线程寄存器减少到 64 个，则：
- 每 Block 寄存器需求 = 256 × 64 × 4 = 64 KB
- 最大 Block 数 = 256 / 64 = 4 个 Block
- Occupancy = 4 × 8 / 64 = **50%**（提升 2×）

</details>

### 题目 4：线程数限制分析

**问题**：
你编写了一个 kernel，Block 大小为 128 线程。

1. 计算在不考虑寄存器和共享内存的情况下，每 SM 最大 Block 数。
2. 如果每线程使用 128 个寄存器，重新计算。
3. 如果每线程使用 255 个寄存器（最大值），重新计算。

<details>
<summary>点击查看答案</summary>

**解答**：

**情况 1：仅考虑线程数限制**
```
每 Block 线程 = 128
每 SM 最大线程 = 2048
最大 Block 数 = 2048 / 128 = 16 个 Block

Occupancy = 16 × (128/32) / 64 = 16 × 4 / 64 = 100%
```

**情况 2：每线程 128 个寄存器**
```
每线程寄存器 = 128 × 4 = 512 bytes
每 Block 需要 = 128 × 512 = 65,536 bytes = 64 KB

寄存器限制 Block 数 = 256 KB / 64 KB = 4 个 Block
线程限制 Block 数 = 16 个 Block

取最小值 = 4 个 Block
Occupancy = 4 × 4 / 64 = 25%
```

**情况 3：每线程 255 个寄存器**
```
每线程寄存器 = 255 × 4 = 1,020 bytes
每 Block 需要 = 128 × 1,020 = 130,560 bytes ≈ 127.5 KB

寄存器限制 Block 数 = 256 KB / 127.5 KB ≈ 2 个 Block
（实际 CUDA 编译器会限制为 2 个 Block）

Occupancy = 2 × 4 / 64 = 12.5%
```

**结论**：过高的寄存器使用会严重降低 Occupancy！

</details>

### 题目 5：GEMM 场景综合计算

**问题**：
你在 RTX 5090 上优化一个 SGEMM kernel，考虑以下三种配置方案：

**方案 A（Shared Kernel 风格）**：
- Block: 32×32 = 1024 线程
- 共享内存：sA[32][32], sB[32][32] = 8 KB
- 寄存器：约 32 个/线程

**方案 B（Register Kernel 风格）**：
- Block: 16×16 = 256 线程
- 共享内存：sA[128][8], sB[8][128] = 8 KB
- 寄存器：约 90 个/线程

**方案 C（优化尝试）**：
- Block: 16×16 = 256 线程
- 共享内存：sA[128][16], sB[16][128] = 16 KB
- 寄存器：约 128 个/线程（TM=16, TN=8）

**任务**：
1. 分别计算三种方案的 Occupancy。
2. 分析哪种方案的 Occupancy 最高。
3. 讨论高 Occupancy 是否一定意味着高性能。

<details>
<summary>点击查看答案</summary>

**解答**：

**方案 A 计算**：
```
线程限制: 2048 / 1024 = 2 Block
寄存器限制: 256KB / (1024×32×4B) = 2 Block
共享内存限制: 164KB / 8KB = 20 Block

最大 Block 数 = min(2, 2, 20) = 2 Block
Occupancy = 2 × (1024/32) / 64 = 2 × 32 / 64 = 100%
```

**方案 B 计算**：
```
线程限制: 2048 / 256 = 8 Block
寄存器限制: 256KB / (256×90×4B) = 2.84 → 2 Block
共享内存限制: 164KB / 8KB = 20 Block

最大 Block 数 = min(8, 2, 20) = 2 Block
Occupancy = 2 × (256/32) / 64 = 2 × 8 / 64 = 25%
```

**方案 C 计算**：
```
线程限制: 2048 / 256 = 8 Block
寄存器限制: 256KB / (256×128×4B) = 2 Block
共享内存限制: 164KB / 16KB = 10 Block

最大 Block 数 = min(8, 2, 10) = 2 Block
Occupancy = 2 × 8 / 64 = 25%
```

**结果对比**：

| 方案 | Occupancy | 限制因素 |
|------|-----------|----------|
| A | **100%** | 线程+寄存器 |
| B | **25%** | 寄存器 |
| C | **25%** | 寄存器 |

**分析**：

方案 A 的 Occupancy 最高（100%），但这**不代表性能最好**：

1. **方案 A 问题**：
   - 每线程只计算 1 个元素
   - 共享内存访问频繁（32 次/迭代）
   - 实际性能低（约 9 TFLOPS）

2. **方案 B 优势**：
   - Occupancy 仅 25%，但每线程计算 64 个元素
   - 更高的计算强度
   - 更好的寄存器复用
   - 实际性能更高（预计 30-50 TFLOPS）

3. **结论**：
   - **高 Occupancy ≠ 高性能**
   - 关键指标是**计算单元利用率**和**内存带宽利用率**
   - 适当的低 Occupancy 配合高每线程计算量可能更好

</details>

### 题目 6：Warp 调度分析

**问题**：
假设一个 kernel 配置如下：
- Block 大小：256 线程 = 8 Warps
- 每 SM 同时运行 4 个 Block = 32 Warps
- 每个 Warp 每轮计算需要 4 个时钟周期
- 全局内存延迟：400 个时钟周期

1. 计算需要多少个 Warp 才能完全隐藏全局内存延迟？
2. 当前配置能否完全隐藏延迟？
3. 如果不能，需要如何调整？

<details>
<summary>点击查看答案</summary>

**解答**：

1. **隐藏延迟所需 Warp 数**：
   ```
   延迟隐藏公式：
   所需 Warp 数 = 内存延迟 / 每 Warp 执行时间
              = 400 / 4
              = 100 Warps
   
   或考虑 4 个 Warp 调度器：
   所需 Warp 数 = 100 / 4 = 25 Warps（同时执行的）
   ```

2. **当前配置评估**：
   ```
   当前活跃 Warp 数 = 32 Warps
   所需 Warp 数 = 25 Warps
   
   32 > 25，所以可以完全隐藏延迟 ✓
   ```

3. **如果不能隐藏的调整方案**：
   ```
   如果活跃 Warp 数只有 16 Warps：
   - 方案 1：增加 Block 数
     每 SM 运行 8 Block = 64 Warps
   
   - 方案 2：减少每线程寄存器使用
     从 128 寄存器降到 64 寄存器
     Block 数从 2 增加到 4
     Warp 数从 16 增加到 32
   
   - 方案 3：使用异步加载指令 (cp.async)
     重叠计算和数据传输
   ```

**关键点**：
- 通常需要至少 16-32 个活跃 Warp 才能较好隐藏延迟
- Occupancy 只是参考，关键是活跃 Warp 数
- RTX 5090 每 SM 最多 64 Warps，通常建议保持 50% 以上

</details>

### 题目 7：动态分区分析

**问题**：
RTX 5090 的共享内存和 L1 缓存共享 164 KB 空间，可以动态配置：

| 配置模式 | 共享内存 | L1 缓存 |
|---------|---------|--------|
| 默认 | 128 KB | 36 KB |
| 最大共享内存 | 164 KB | 0 KB |
| 平衡 | 96 KB | 68 KB |

假设你的 kernel 使用 48 KB 共享内存，分析：

1. 三种配置下每 SM 最大 Block 数。
2. 哪种配置最适合你的 kernel？
3. 如果你的 kernel 使用 160 KB 共享内存，会发生什么？

<details>
<summary>点击查看答案</summary>

**解答**：

**情况 1：48 KB 共享内存使用**

```
默认配置 (128 KB 共享内存):
最大 Block 数 = 128 / 48 = 2.67 → 2 Block

最大共享内存配置 (164 KB):
最大 Block 数 = 164 / 48 = 3.42 → 3 Block

平衡配置 (96 KB):
最大 Block 数 = 96 / 48 = 2 Block
```

**最佳选择**：最大共享内存配置（3 个 Block vs 2 个 Block）

配置代码：
```cuda
cudaFuncSetAttribute(kernel, cudaFuncAttributePreferredSharedMemoryCarveout, 
                     cudaSharedmemCarveoutMaxShared);
```

**情况 2：160 KB 共享内存使用**

```
默认配置: 128 KB < 160 KB → 无法运行！
最大配置: 164 KB > 160 KB → 只能运行 1 Block

问题：
- kernel 无法启动，或
- 编译错误，或
- 运行时 CUDA 错误

解决方案：
1. 减少共享内存使用到 ≤ 164 KB
2. 使用动态共享内存分配
3. 减小 Block 大小或分块大小
```

</details>

## 综合练习题

### 题目 8：设计最优配置

**问题**：
你希望在 RTX 5090 上实现一个高性能 SGEMM kernel，目标性能为 40 TFLOPS（FP32）。

已知：
- RTX 5090 FP32 峰值：104.9 TFLOPS
- 目标利用率：40 / 104.9 ≈ 38%
- 每 SM 寄存器：256 KB
- 每 SM 共享内存：164 KB

**任务**：
1. 设计一组合理的 Block 大小、TM/TN、BK 参数。
2. 计算该配置的 Occupancy。
3. 分析该配置是否能达到目标性能。

**提示**：
- 考虑寄存器使用在 64-128 之间
- 共享内存使用在 8-32 KB 之间
- 每线程计算量需要足够大

<details>
<summary>点击查看答案</summary>

**参考设计方案**：

**方案 A：高 Occupancy 路线**
```
Block: 16×16 = 256 线程
TM=8, TN=8 → 每线程 64 寄存器
BK=8 → 共享内存 8 KB

寄存器限制: 256KB / (256×64×4B) = 4 Block
共享内存限制: 164KB / 8KB = 20 Block
线程限制: 2048 / 256 = 8 Block

最大 Block = min(4, 20, 8) = 4 Block
Occupancy = 4 × 8 / 64 = 50%

每 SM 计算量: 4 Block × 256 线程 × 8×8 FMA = 65,536 FMA
预计性能: 30-40 TFLOPS ✓
```

**方案 B：高计算强度路线**
```
Block: 16×16 = 256 线程
TM=16, TN=8 → 每线程 128+16=144 寄存器 (估算)
BK=16 → 共享内存 16 KB

寄存器限制: 256KB / (256×144×4B) = 1.74 → 1 Block
（可能需要减少到 128 寄存器）

调整为 TM=16, TN=8, 优化后 128 寄存器:
寄存器限制: 2 Block
Occupancy = 2 × 8 / 64 = 25%

但每线程计算量翻倍：16×8=128 FMA/迭代
预计性能: 40-50 TFLOPS ✓（如果代码效率足够高）
```

**推荐**：方案 A（50% Occupancy + 足够计算量）

</details>

## 公式速查表

### Occupancy 计算步骤

```
1. 计算各资源限制下的最大 Block 数：

   线程限制 = 2048 / Block 大小
   
   寄存器限制 = 256KB / (Block 大小 × 每线程寄存器 × 4)
   
   共享内存限制 = 164KB / 每 Block 共享内存

2. 取最小值：
   最大 Block = min(线程限制, 寄存器限制, 共享内存限制)

3. 计算 Occupancy：
   Occupancy = (最大 Block × Block 大小 / 32) / 64 × 100%
```

### 常用参数参考

| 配置 | 线程数 | 寄存器/线程 | 共享内存 | 每 SM Block 数 | Occupancy |
|------|--------|------------|---------|---------------|-----------|
| 保守 | 256 | 64 | 8 KB | 4 | 50% |
| 平衡 | 256 | 90 | 8 KB | 2 | 25% |
| 激进 | 128 | 128 | 16 KB | 2 | 25% |
| 最大 | 1024 | 32 | 32 KB | 2 | 100% |

通过完成这些练习题，你应该能够：

1. ✅ 熟练计算 Occupancy

**关键要点**：
- Occupancy 是重要的，但不是唯一指标

- 平衡 Occupancy (25-50%) 和高计算强度通常是最好的
- 寄存器使用是最常见的瓶颈
