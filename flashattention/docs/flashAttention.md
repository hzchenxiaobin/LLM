# 深入浅出理解 Flash Attention

Flash Attention 是由 Tri Dao 等人在 2022 年提出的一种计算自注意力（Self-Attention）的算法。可以说，它是近年来大模型（LLM）能够支持超长上下文（Context Window，如 32k、128k 甚至 1M）的最核心基石之一。

理解 Flash Attention，最关键的是要明白：它不是通过改变数学公式来近似计算，而是通过「硬件感知（Hardware-aware）」来优化计算过程，实现了一字不差的精确计算（Exact Attention），同时大幅提升了速度并降低了显存占用。

## 1. 痛点：标准 Attention 为什么慢？

在标准的 Transformer 中，Attention 的计算公式为：

```
Attention(Q, K, V) = softmax(Q × K^T / √d) × V
```

假设序列长度为 N，特征维度为 d。标准计算过程包含以下几步：

1. 计算 S = Q × K^T（复杂度 O(N²)）
2. 计算 P = softmax(S)（复杂度 O(N²)）
3. 计算 O = P × V（复杂度 O(N²)）

真正的问题不在于计算量（FLOPs），而在于「内存墙（Memory Wall）」。

GPU 的存储分为两级：

- **HBM（高带宽内存 / 显存）**：容量大（如 80GB），但读写速度相对较慢。
- **SRAM（片上缓存）**：容量极小（如 20MB），但在计算核心旁边，读写速度极快。

### 图示：标准 Attention 的内存噩梦

在标准 Attention 中，巨大的 O(N²) 中间矩阵 S 和 P 必须被来回搬运：

```text
[ HBM (慢速, 容量大) ]                      [ SRAM (快速, 容量小) ]
   Q, K ----------------- 读取 ------------> 计算 S = Q*K^T
   S <------------------- 写入 -------------
   S -------------------- 读取 ------------> 计算 P = Softmax(S)
   P <------------------- 写入 -------------
   P, V ----------------- 读取 ------------> 计算 O = P*V
   O <------------------- 写入 -------------
```

每一次中间结果的读写，都是 O(N²) 的数据量。GPU 的计算单元实际上在「干等」数据的搬运。

**通俗比喻**：想象你在一个很小的厨房（SRAM）做饭，但所有的食材都在地下室的大仓库（HBM）里。标准做法是：你把所有的肉（Q 和 K）搬到厨房切好，做成半成品（S 矩阵），然后把半成品全部搬回地下室。接着为了调味（Softmax），你又把半成品全搬上来，调完味（P 矩阵）再搬回去。最后再搬上来加蔬菜（V）炒熟（O）。绝大部分时间都浪费在了上下楼梯搬东西上。

## 2. Flash Attention (V1) 的三大核心创新

Flash Attention 的目标是：尽可能把数据留在 SRAM 中计算完毕，避免中间结果（S 和 P）在 HBM 和 SRAM 之间来回搬运。

### 图示：Flash Attention 的优雅流程

Flash Attention 将庞大的矩阵切分成小块（Blocks），在 SRAM 中完成全部计算，最后只把结果 O 写回：

```text
[ HBM ]                                     [ SRAM ]
 Q,K,V 块 ---- 分块读取 (Tiling) ----------> | 1. 计算局部 S_block
                                            | 2. 更新 Online Softmax
                                            | 3. 计算局部 O_block
   O_最终 <---- 仅写入最终结果 O_block -------+ 
```

### 核心一：分块计算 (Tiling)

既然 N × N 的矩阵太大，SRAM 装不下，那就切块（Block）。

Flash Attention 将 Q、K、V 矩阵切成一个个小块，每次只把一小块数据从 HBM 载入 SRAM，在 SRAM 内完成矩阵乘法和 Softmax 操作，最后把最终结果 O 的一小块写回 HBM。

显存占用从 O(N²) 直接降到了 O(N)。

### 核心二：在线 Softmax (Online Softmax)

分块计算遇到了一个巨大的数学障碍：Softmax 是全局操作。

如果不知道全局信息，就没法算出正确的概率。Online Softmax 技巧允许我们在遍历数据块时，维护并更新两个局部变量：局部最大值 m 和局部指数和 l。

当读入新的数据块时，用新的局部最大值去「修正」之前计算过的结果。这样即使每次只看一部分数据，最后得到的结果也和全局 Softmax 算出来的一模一样。

#### Online Softmax 的数学推导

下面只写**单行**（Attention 里固定一行 q，对整列 key 下标求 softmax）的情形；矩阵形式是对每一行独立重复同一套递推。

**1. 减最大值与归一化**

对 logits 向量 **x** ∈ R^C，标准 softmax 为

```
softmax(x)_i = e^(x_i) / Σ(j=1 to C) e^(x_j)
```

任取常数 m，都有恒等变形

```
softmax(x)_i = e^(x_i - m) / Σ(j=1 to C) e^(x_j - m)
```

取 m = max_j(x_j)，则所有指数 x_j - m ≤ 0，既避免溢出又便于分块时统一「基准」。

记当前行的**指数和**（分母）为

```
L = Σ(j=1 to C) e^(x_j - m)
```

**2. 两段合并：新的全局最大值**

把下标分成先后两块（对应先看到一块 K、再看到下一块 K）。设已处理集合上的最大值为 m^old，指数为

```
L^old = Σ(j∈旧块) e^(x_j - m^old)
```

新来一块 logits，在该块内行内最大值为 m^block。定义**新的全局最大值**

```
m^new = max(m^old, m^block)
```

旧块里任意一项在新基准下应写成 e^(x_j - m^new) = e^(x_j - m^old) × e^(m^old - m^new)，因此旧块对指数和的贡献整体乘上 e^(m^old - m^new)。新块直接按 m^new 写指数。合并后

```
L^new = L^old × e^(m^old - m^new) + Σ(j∈新块) e^(x_j - m^new)
```

这正是伪代码里「先缩放旧的 l，再加上新块的 Σ exp」的来源。多块时从左到右反复套用同一式，即得**在线**更新，无需事先看到全部 C 个 logits。

**3. 与 V 的加权求和 o = Σ_i softmax_i × v_i**

定义未归一化的分子（向量）

```
õ = Σ(j) e^(x_j - m) × v_j
```

则 o = õ / L。当 m 从 m^old 更新到 m^new 时，旧贡献同样整体乘以 e^(m^old - m^new)，新块直接加上 Σ(j∈新块) e^(x_j - m^new) × v_j（矩阵形式即 P_ij × V_j）。故

```
õ^new = e^(m^old - m^new) × õ^old + Σ(j∈新块) e^(x_j - m^new) × v_j
```

若每一步都对 õ/L 做归一化，就得到前文 V1 伪代码里「每一步除以 l_new」的写法；若只在整行所有块处理完后做一次 o = õ/L，则对应 V2 把除法**推迟到行末**的写法——两者数学相同，只是缩放时机不同。

**4. 小结**

Online Softmax 的本质是：只维护 (m, L, õ) 三个量，每来一块 logits 就用新的 max 重定基准，对旧量做一次**指数 rescaling**，再叠加上新块的贡献；最后再 o = õ/L。这样全程无需在内存里存放完整的 N×N 的 S 或 P。

### 核心三：反向传播的重计算 (Recomputation)

为了节省显存，前向传播时我们根本没有保存庞大的中间矩阵（S 和 P）。

反向传播怎么算梯度？重计算。反向传播时，利用保留的 Q、K、V 重新分块计算一次局部的 Softmax 结果，然后再算梯度。因为避免了 HBM 的海量数据读取，即便是重新计算一遍，总时间也比去 HBM 里读数据快得多。

## 3. 伪代码对比：体会 SRAM 中的魔法

### 传统 Attention 伪代码

```python
def standard_attention(Q, K, V):
    S = Q @ K.transpose() # 写入 HBM
    P = softmax(S)        # 从 HBM 读 S，将 P 写入 HBM
    O = P @ V             # 从 HBM 读 P 和 V，将 O 写入 HBM
    return O, S, P        # 保存 S 和 P 用于反向传播计算梯度
```

### Flash Attention V1 简化伪代码

```python
def flash_attention_v1(Q, K, V, block_size_M, block_size_N):
    O = zeros(N, d)
    l = zeros(N, 1) # 全局指数和
    m = fill(N, 1, -inf) # 全局最大值

    # 外层循环：遍历 K, V 
    for j in range(0, N, block_size_N):
        K_j = K[j : j + block_size_N]
        V_j = V[j : j + block_size_N]

        # 内层循环：遍历 Q 
        for i in range(0, N, block_size_M):
            Q_i = Q[i : i + block_size_M]
            O_i, l_i, m_i = O[i...], l[i...], m[i...]

            # 1. 局部矩阵乘法 (SRAM)
            S_ij = Q_i @ K_j.transpose()

            # 2. Online Softmax 更新 (SRAM)
            m_ij = max(S_ij, axis=1)
            m_new = max(m_i, m_ij)   # 修正后的全局最大值
            
            P_ij = exp(S_ij - m_new)
            l_new = l_i * exp(m_i - m_new) + sum(P_ij, axis=1)

            # 3. 更新局部输出 (SRAM) -> 注意这里有繁琐的缩放操作
            O_i = (O_i * l_i * exp(m_i - m_new) + P_ij @ V_j) / l_new

            # 写回 HBM
            m[i...], l[i...], O[i...] = m_new, l_new, O_i
    return O
```

## 4. Flash Attention V1 的底层并行架构：SM、Warp、线程分配

理解 Flash Attention V1 的 GPU 底层实现，需要了解 NVIDIA GPU 的三级执行架构：

- **SM（Streaming Multiprocessor）**：GPU 的计算核心单元（如 A100 有 108 个 SM），每个 SM 能并发执行多个线程块（Block）。
- **Warp**：一个 Warp 包含 32 个线程，是 GPU 调度的基本单位。一个 Warp 内的线程执行相同的指令（SIMT 架构）。
- **线程（Thread）**：最基本的执行单元，每个线程处理一个或少量数据元素。

### 4.1 SM 级别的任务分配

在 Flash Attention V1 中，SM 级别的并行主要基于 **Batch 维度** 和 **Head 维度**：

```text
SM 任务分配图（FlashAttention V1）
====================================

┌─────────────────────────────────────────────────────────────┐
│                        GPU 全局内存 (HBM)                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐       │
│  │ Batch 0  │ │ Batch 1  │ │ Batch 2  │ │ Batch 3  │       │
│  │ Head 0   │ │ Head 0   │ │ Head 0   │ │ Head 0   │       │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘       │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│   SM 0        │    │   SM 1        │    │   SM 2        │
│ ┌───────────┐ │    │ ┌───────────┐ │    │ ┌───────────┐ │
│ │Block(0,0) │ │    │ │Block(1,0) │ │    │ │Block(2,0) │ │
│ │Batch 0    │ │    │ │Batch 1    │ │    │ │Batch 2    │ │
│ │Head 0     │ │    │ │Head 0     │ │    │ │Head 0     │ │
│ └───────────┘ │    │ └───────────┘ │    │ └───────────┘ │
└───────────────┘    └───────────────┘    └───────────────┘

每个 SM 处理一个 (Batch, Head) 组合
不同 SM 之间完全独立，无通信开销
```

**分配策略**：
- 每个 CUDA Block（线程块）被分配到一个 SM 上执行
- 一个 (Batch, Head) 组合对应一个或多个 CUDA Block
- 在序列长度维度上，V1 **不进行**跨 SM 的并行（这是 V2 的改进点）

### 4.2 Warp 级别的任务分配

在一个 SM 内部，Flash Attention V1 将任务分配给多个 Warp。这是 V1 实现的关键细节：

```text
单个 SM 内部：Warp 分工处理 Q、K、V 块
================================================

外层循环：遍历 K_j, V_j 块（列方向，沿序列长度 N）
    │
    │   ┌───────────────────────────────────────────────┐
    │   │              共享内存 (SRAM)                   │
    │   │  ┌─────────────────────────────────────────┐  │
    └───┼─▶│           K_j 块  [Bc × d]              │  │
        │  │  ┌─────────────────────────────────────────┐│
        └──┼─▶│           V_j 块  [Bc × d]              ││
           │  └─────────────────────────────────────────┘│
           └───────────────────────────────────────────────┘
                              │
                              ▼
内层循环：遍历 Q_i 块（行方向，每个 Warp 处理一个子块）
    │
    ├───▶ Warp 0 ──▶ 处理 Q_i[0:16]  ──▶ 计算 S[0:16, :] ──▶ 更新 m,l,O
    │
    ├───▶ Warp 1 ──▶ 处理 Q_i[16:32] ──▶ 计算 S[16:32, :] ──▶ 更新 m,l,O
    │
    ├───▶ Warp 2 ──▶ 处理 Q_i[32:48] ──▶ 计算 S[32:48, :] ──▶ 更新 m,l,O
    │
    └───▶ Warp 3 ──▶ 处理 Q_i[48:64] ──▶ 计算 S[48:64, :] ──▶ 更新 m,l,O

    ... 多个 Warp 协作处理一个 Q_i 块

⚠️ 关键问题：Warp 之间需要同步 m、l、O
```

**Warp 分工的核心机制**：

1. **K、V 块的协作加载**：
   - 所有 Warp 协作将 K_j 和 V_j 从 HBM 加载到共享内存（SRAM）
   - 每个 Warp 负责加载一部分行或列

2. **Q 块的分割处理**：
   - Q_i 块被分割成更小的子块，每个 Warp 处理一个子块
   - 例如：Q_i 大小为 `[Br, d]`，4 个 Warp 各处理 `[Br/4, d]`

3. **矩阵乘法的 Warp-level 并行**：
   - `S_ij = Q_i @ K_j.T` 使用 Tensor Core 加速
   - Warp 使用 WMMA（Warp Matrix Multiply Accumulate）指令协作计算

### 4.3 线程级别的任务分配

在一个 Warp 内部，32 个线程进一步分工：

```text
Warp 内部：32 线程的 SIMD 执行
================================

Warp 处理 Q_i 的一个子块 [Br_sub × d] = [16 × 64]（示例）

线程 0-7:   处理 Q_i[0:2, :]  的 2 行，每行用 4 线程协作
线程 8-15:  处理 Q_i[2:4, :]  的 2 行
线程 16-23: 处理 Q_i[4:6, :]  的 2 行  
线程 24-31: 处理 Q_i[6:8, :]  的 2 行

每个线程执行（以线程 0 为例）：
─────────────────────────────────────
for k_idx in range(0, Bc, 4):      # Bc = K_j 的块大小
    # 加载 K_j 的 4 个元素到寄存器
    k0, k1, k2, k3 = K_j[k_idx:k_idx+4]
    
    # 计算点积的一部分（FMA 指令）
    acc += q_vec[0] * k0
    acc += q_vec[1] * k1
    acc += q_vec[2] * k2
    acc += q_vec[3] * k3
─────────────────────────────────────

所有 32 线程同时执行相同的指令流，处理不同数据
```

### 4.4 V1 的关键瓶颈：Warp 间同步

Flash Attention V1 的核心问题出在 **Warp 间的频繁同步**：

```text
V1 的 Warp 同步模式（问题所在）
=================================

外层：加载 K_j, V_j ──────────────────────────┐
    │                                          │
    ▼                                          │
内层：对每个 Q_i 块                            │
    │                                          │
    ├── Warp 0 计算 S[0:16, :]                  │
    │   ├── 计算局部 m_0（行内最大值）         │
    │   ├── 需要知道 Warp 1/2/3 的 m_1/2/3     │
    │   ├── 🚨 同步点：共享内存读写 m            │
    │   └── 更新全局 m_new                      │
    │                                          │
    ├── Warp 1 计算 S[16:32, :]                │
    │   ├── 同样需要等待其他 Warp 的 m          │
    │   ├── 🚨 同步点：共享内存读写 m            │
    │   └── 更新全局 m_new                      │
    │                                          │
    ├── 🔄 所有 Warp 同步完成才能继续           │
    │                                          │
    └── 🚨 下一轮 Q_i 又要重新同步              │
                                               │
    ◀──────────────────────────────────────────┘
    
问题：外层每迭代一次 K/V，内层所有 Warp 都要同步多次！
```

**同步的具体内容**：

每个 Warp 在内层循环中需要维护并同步三个量：
1. **m（行最大值）**：需要通过 Warp Shuffle 或共享内存比较全局最大值
2. **l（指数和）**：需要根据新的 m 进行缩放并累加
3. **O（输出累加器）**：需要根据新的 m 进行重缩放

```python
# V1 中每个 Warp 在内层循环必须执行的同步操作（简化示意）

# 假设当前 Warp 计算得到了局部信息
m_local = max(S_ij, axis=1)      # 局部行最大值
m_new = warp_reduce_max(m_local) # 🚨 跨 Warp 求全局 max（通过共享内存）

# 缩放旧的累加器
scale = exp(m_old - m_new)       # 每个线程计算
l_i *= scale                     # 需要确保所有 Warp 拿到相同的 m_new
O_i *= scale                     # 同样依赖同步后的 m_new

# 累加新的贡献
l_i += sum(exp(S_ij - m_new))    # 局部求和后再次同步
O_i += P_ij @ V_j               # 更新输出
```

### 4.5 共享内存（SRAM）的使用布局

Flash Attention V1 精心规划了共享内存的使用：

```text
单个 Block 的共享内存布局（A100，约 164KB 可用）
===================================================

┌──────────────────────────────────────────────────┐
│              共享内存分配（约 40-60KB 使用）        │
├──────────────────────────────────────────────────┤
│  K_j 缓冲区  │  [Bc × d]  │  16KB (Bc=64, d=64)   │
├──────────────────────────────────────────────────┤
│  V_j 缓冲区  │  [Bc × d]  │  16KB                 │
├──────────────────────────────────────────────────┤
│  Q_i 缓冲区  │  [Br × d]  │  16KB (Br=64)         │
├──────────────────────────────────────────────────┤
│  S_ij 缓冲区 │  [Br × Bc] │  16KB (用于中间结果)   │
├──────────────────────────────────────────────────┤
│  m, l 共享区 │  [Br] 每个 │  较小，用于 Warp 同步  │
│  （全局 max  │  8 Bytes   │                       │
│   和累加器） │            │                       │
└──────────────────────────────────────────────────┘

寄存器使用（每个线程）：
- Q 的子向量、K 的子向量、累加器
- 临时计算结果（如 exp 值）
```

### 4.6 V1 分配策略总结

| 层级 | 分配单元 | 分配策略 | 通信模式 |
|------|----------|----------|----------|
| **SM** | CUDA Block | 按 (Batch, Head) 分配 | 无通信，完全独立 |
| **Warp** | Q_i 子块 | 每个 Warp 处理 Q_i 的一部分行 | 频繁同步 m, l, O |
| **线程** | 具体元素 | 每线程处理 d 维向量的子集 | Warp 内 SIMD 执行 |

**V1 的核心痛点**：
- 由于外层循环遍历 K/V，内层循环遍历 Q，导致每个 K/V 块迭代时，所有 Warp 都要重新同步
- Warp 间的 m、l、O 同步通过共享内存完成，产生大量读写冲突
- A100 GPU 的 Tensor Core 利用率仅 25%~40%，大量时间花在非矩阵乘法操作上

这些痛点正是 Flash Attention-2 要解决的问题——通过调换循环顺序，让 Warp 独占一整行 Q 的计算，几乎消除 Warp 间同步。

## 6. 进阶：Flash Attention 2 到底优化了什么？

如果说 Flash Attention V1 的核心是**「打破内存墙」，把瓶颈从访存变成了计算**；那么 Flash Attention-2 的核心就是**「极致压榨计算力」**。

在 V1 时代，虽然速度快了，但人们发现 A100 GPU 上的 Tensor Core（专门做矩阵乘法的硬件）利用率只有 25%～40%。V2 的目标就是让 Tensor Core 跑满，最终将利用率提升到了惊人的 70%～73%，整体速度又翻了约一倍。

它做了三大核心升级：

### 升级一：减少非矩阵乘法开销 (Fewer non-matmul FLOPs)

仔细看上面 V1 的伪代码中 O_i 的更新公式：(O_i ⊙ l_i ⊙ exp(m_i - m_new) + P_ij × V_j) / l_new。

每一次内层循环，都要进行大量的除法、乘法和指数运算。这些**非矩阵乘法（Non-matmul）**运算不能由 Tensor Core 执行，只能由普通的 CUDA Core 执行，速度慢，严重拖慢了整体节奏。

**V2 的解法**：通过代数变换，推迟缩放（Rescaling）。在内层循环中，它只做加法和矩阵乘法，把所有需要除以分母（指数和）的操作，全部推迟到整行块计算完毕后，在最外层循环仅仅执行一次。这让 Tensor Core 得以火力全开。

### 升级二：序列维度的并行化 (Sequence Parallelism)

在 V1 中，并行化的单位是 Batch（批次）和 Head（注意力头）。系统把不同的 Batch 和 Head 分配给 GPU 上不同的计算单元（SM，流多处理器）。

**痛点**：如今处理超长文本（Long Context）时，Batch Size 往往是 1，Head 数量也是固定的（比如 32）。而一块 A100 GPU 有 108 个计算单元，这就导致大量计算单元处于闲置状态（即「饿肚子」）。

**V2 的解法**：在 Sequence Length（序列长度）维度上也进行切分并行。即使 Batch = 1，长达 100k 的序列也会被切成许多小块，均匀分发给所有的 GPU 计算单元。这极大提升了长文本场景下的 GPU 占用率。

### 升级三：优化循环顺序与共享内存同步 (Better Work Partitioning)

在一个 GPU 计算单元内部，有多个更小的执行线程组（称为 Warp）。它们共用一块 SRAM（Shared Memory）。

**V1 的循环结构**：外循环遍历 K、V，内循环遍历 Q。这意味着在计算一小块数据时，不同的 Warp 之间需要频繁地把局部的 l_i 和 m_i 写入 SRAM 进行同步和共享。

**V2 的循环结构颠倒**：外循环遍历 Q，内循环遍历 K、V。

这样调整后，系统可以将一个巨大的 Q 块固定分配给特定的 Warp。这个 Warp 只需要自己闷头遍历 K 和 V 就能完成整行的 Attention 计算。Warp 之间不再需要频繁通信，大幅减少了 SRAM 的读写冲突（Shared Memory Traffic）。

### 图示：V1 与 V2 的循环顺序对比

```text
FlashAttention-1                          FlashAttention-2
─────────────────                         ─────────────────
for j:  K_j, V_j 块                         for i:  Q_i 块
    for i:  Q_i 块      ← 内层频繁换 Q           for j:  K_j, V_j 块  ← 内层扫完整条 KV
        Online softmax                         Online softmax
        (Warp 间要同步 m, l, O)                  (每个 Warp 独占一行块的 m, l, O)
```

### 图示：FlashAttention-2 的数据流（固定 Q 块、顺序扫 K/V）

同一 Q 块驻留在 SRAM/寄存器中，沿序列维依次消费各 K、V 块；整行 Online Softmax 完成后，仅对输出做一次按行除法（归一化），减少内层循环里的标量除法。

```text
[ HBM ]                                              [ SRAM / 寄存器 ]
                                                     
  Q_i 块 ---- 一次加载，外层固定 ------------------>  Q_i 常驻
                                                      |
  K_j,V_j ---- j=0,1,2,... 内层依次加载 ----------->  S_ij = Q_i K_j^T
                                                      |    Online: 更新 m, l
                                                      |    累加 O_i（先不除 l）
                                                      v
  O[i:...] <--- 内层 j 结束后写回 ------------------  O_i ← O_i / l_i  （每行仅一次除法）
```

### 伪代码：FlashAttention-2（循环交换 + 行末归一化）

下面与 V1 使用相同的 Online Softmax 数学，但**外层遍历 Q 块、内层遍历 K/V 块**；**把对输出的除法推迟到该 Q 块处理完所有 K/V 之后**，从而减少内层循环中的非矩阵乘法开销。

```python
def flash_attention_v2(Q, K, V, Br, Bc):
    """
    Q, K, V: [N, d]
    Br: Q 方向块大小（行块）；Bc: K/V 方向块大小（列块）
    """
    N, d = Q.shape
    O = zeros(N, d)

    # 外层：Q 块 —— 与 V1 相反；便于 Warp 绑定行块、减少同步
    for i in range(0, N, Br):
        Qi = Q[i : i + Br]                    # [Br, d]
        Oi = zeros(Br, d)
        li = zeros(Br)
        mi = full(Br, -inf)

        # 内层：K,V 块 —— 顺序扫过整段序列
        for j in range(0, N, Bc):
            Kj = K[j : j + Bc]
            Vj = V[j : j + Bc]

            S_ij = Qi @ Kj.T                  # [Br, Bc]  GEMM，走 Tensor Core
            m_row = max(S_ij, axis=1)         # 本块行内 max
            m_new = maximum(mi, m_row)      # 逐元素 max，与历史全局 max 合并

            # 数值稳定的 exp；P_ij 尚未除以最终 softmax 分母
            P_ij = exp(S_ij - m_new[:, None])
            li = li * exp(mi - m_new) + sum(P_ij, axis=1)
            Oi = Oi * exp(mi - m_new)[:, None] + P_ij @ Vj

            mi = m_new

        # 该 Q 块对应的所有 K/V 块处理完毕后再归一化（减少内层除法）
        O[i : i + Br, :] = Oi / li[:, None]

    return O
```

说明：`maximum`、`exp`、`sum` 等为按行向量运算；真实实现还会融合掩码、变长序列、scale = 1/√d 等，且 l_i 会加极小量避免除零。序列维并行体现在：不同的 i 区间（不同 Q 块）可派发到不同 SM/线程块同时执行。

## 7. Flash Attention V2 的底层并行架构：SM、Warp、线程分配

Flash Attention V2 不仅改变了算法逻辑（调换循环顺序），更重要的是彻底重构了 GPU 三级并行架构的任务分配策略。这让 V2 能够榨干 Tensor Core 的算力，将 A100 利用率从 V1 的 25%~40% 提升到 70%~73%。

### 7.3 线程级别的任务分配：更高效的寄存器使用

V2 在线程级别的分配与 V1 类似，但由于 Warp 独占整行 Q，寄存器使用更加高效：

```text
V2 线程级别数据流（单个 Warp 内部）：
======================================

Warp 处理 Q_i 的 4 行，每行 d=64

线程 0-7  : 负责行 0 的前 8 维（vec[0:8]）    
线程 8-15 : 负责行 0 的中 8 维（vec[8:16]）
线程 16-23: 负责行 0 的后 8 维（vec[16:24]）
线程 24-31: 负责行 0 的最后 8 维（vec[24:32]）

注意：这里假设 d=64，每个线程负责 2 个元素（使用 float2 或 half2 向量化）

内层循环 j（以线程 0 为例）：
─────────────────────────────────────
# 寄存器分配（每个线程）
reg_q[2]      # Q 行向量的 2 个元素
reg_m         # 行最大值（标量，线程 0 代表行保存）
reg_l         # 行指数和（标量，线程 0 代表行保存）  
reg_o[2]      # 输出行累加器的 2 个元素
reg_s[Bc]     # 临时分数向量（Bc=64，可能用共享内存缓冲）

for j in range(0, N, Bc=64):
    # 协作加载 K_j 到共享内存
    load_Kj_to_smem()
    
    # 矩阵乘：q @ K_j.T（使用 Tensor Core WMMA）
    # 每个线程计算 S[0:4, j:j+Bc] 的一部分
    mma_sync()  # Warp-level 矩阵乘
    
    # 获取当前块的最大值（Warp reduce）
    m_block = warp_reduce_max(reg_s)
    
    # 更新全局最大值（寄存器私有，无需跨 Warp）
    m_old = reg_m
    reg_m = max(reg_m, m_block)
    
    # 缩放旧的累加器
    scale = exp(m_old - reg_m)
    reg_l *= scale
    reg_o[0] *= scale
    reg_o[1] *= scale
    
    # 累加新的贡献
    for k_idx in range(Bc):
        p = exp(reg_s[k_idx] - reg_m)
        reg_l += p
        # P @ V 的一部分
        reg_o[0] += p * Vj[k_idx, 0]
        reg_o[1] += p * Vj[k_idx, 1]

# 循环结束后，归一化
reg_o[0] /= reg_l
reg_o[1] /= reg_l
write_to_HBM()
```

**V2 线程级优化的关键点**：

1. **m、l、O 完全寄存器化**：由于 Warp 独占一行，m、l、O 可以全程保存在寄存器中，不需要写入共享内存
2. **减少寄存器压力**：推迟除法到行末，内层循环只需要乘法和加法
3. **向量化访存**：使用 float4 或 half8 加载 Q、K、V，提升内存带宽利用率

### 7.4 共享内存（SRAM）的使用优化

V2 的共享内存使用比 V1 更高效，因为不需要为 Warp 同步预留 m、l 缓冲区：

```text
V2 共享内存布局 vs V1 对比：
=============================

│ V_j [Bc × d]      16KB  │                │ V_j [Bc × d]      16KB  │
│ Q_i [Br × d]      16KB  │                │ Q_i [Br × d]      16KB  │
│ S_ij [Br × Bc]    16KB  │                │ S_ij [Br × Bc]    16KB  │
│ m 共享区 [Br]       1KB  │◀── 额外开销    │ （可选临时缓冲）     4KB │
│ l 共享区 [Br]       1KB  │◀── 额外开销    └─────────────────────────┘
│ O 临时区 [Br×d]    16KB  │◀── 额外开销      总计：52KB（可更大块）
└─────────────────────────┘
  总计：66KB                                    
                                              或：减少共享内存使用
```

**V2 的 SRAM 优化策略**：

1. **m、l 完全寄存器化**：每个 Warp 私有，不占用共享内存
2. **Q_i 可以部分寄存器化**：如果 Br 较小（如 64），Q_i 可以直接常驻寄存器
3. **增大 K/V 块大小**：释放的共享内存可以用来增大 Bc，减少外层循环次数

### 7.5 V1 vs V2 三级架构对比总结

| 层级 | V1 策略 | V2 策略 | V2 改进效果 |
|------|---------|---------|-------------|
| **SM 分配** | 仅 Batch × Head 并行 | 增加 Sequence 并行 | 长文本 GPU 利用率从 <30% 提升到 >90% |
| **Warp 分配** | 分割 Q 块，频繁同步 m,l,O | 独占整行 Q，零同步 | 消除共享内存读写冲突，Tensor Core 利用率 25%→70% |
| **线程分配** | SIMD 执行，共享 m,l | SIMD 执行，私有 m,l | 寄存器使用更高效，减少 bank conflict |
| **循环顺序** | 外 K/V，内 Q | 外 Q，内 K/V | Warp 可以独立完成整行计算 |
| **归一化时机** | 每轮内层循环都除 l_new | 推迟到行末统一除 | 内层循环非 matmul FLOPs 减少 50%+ |

### 7.6 具体代码视角：V2 Kernel 的启动配置

```python
# V2 CUDA Kernel 的典型启动配置（概念性伪代码）

# 问题规模
batch_size = 1
num_heads = 32
seq_len = 32768
head_dim = 64

# V2 块大小配置（A100 优化）
Br = 128   # Q 块行数
Bc = 64    # K/V 块大小（可更大，因为 SRAM 压力减小）

# 序列维并行：总 Block 数
num_seq_blocks = seq_len // Br  # 256

# Kernel 启动配置
threads_per_block = 128  # 4 个 Warp，每个 Warp 32 线程
                         # 或 256 线程 = 8 个 Warp
                         
num_blocks = batch_size * num_heads * num_seq_blocks
# = 1 * 32 * 256 = 8192 个 Block

# 这 8192 个 Block 分发到 A100 的 108 个 SM
# 每个 SM 大约处理 76 个 Block，GPU 完全占满

# 每个 Block 内部的 Warp 分配：
warp_per_block = threads_per_block // 32  # 4 或 8 个 Warp
rows_per_warp = Br // warp_per_block      # 每个 Warp 处理 32 或 16 行
```

## 8. 总结：性能演进的阶梯

理解 Flash Attention 及其后续版本，本质上是理解算法与底层硬件（计算机体系结构）的深度绑定：

- **Standard Attention**：算法上最直接，但败给了硬件的「内存墙」（显存读写太慢）。
- **Flash Attention V1**：用 Tiling 和 Online Softmax 拯救了内存读写，把 IO 复杂度降到最低，解决了长文本显存爆炸的问题。
- **Flash Attention V2**：发现内存瓶颈解决后，计算单元的利用率还不高，于是重构循环、减少除法、增加序列并行，把 Tensor Core 的算力榨干。
- **Flash Attention V3 (2024)**：进一步针对英伟达 Hopper 架构（如 H100 GPU）的专属硬件特性（如 TMA 张量内存加速器、WGMMA）进行极致的汇编级优化，速度再次翻倍。

**核心启发**：在现代深度学习中，减少计算量（FLOPs）固然重要，但如何让数据在硬件存储层级中高效流动，往往更能带来数十倍的性能飞跃。
